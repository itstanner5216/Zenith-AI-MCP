Production XML Tool Deferral, Schema Injection, and XML Dispatch for Zenith-MCP
Summary
Build XML-based tool schema deferral for Zenith-MCP using a dual-surface architecture: native JSON tools remain minimal and static, while deferred tools are exposed through prompt-body XML blocks and invoked through XML tool calls. The implementation adds a deterministic XML schema formatter, a SAX-style parser, a virtual registry, per-session retrieval state, host-facing MCP prompts/tools, and an execute_xml/zenith_xml_dispatch loop that routes XML calls to real Zenith handlers. This is not a simplified MVP: it includes cache preservation, reactive retrieval, provider-safe fallbacks, tests, metrics, cleanup, and migration docs.
Locked Decisions
	•	No dynamic mutation of the system prompt. The static XML instruction block is byte-identical across sessions and exposed via zenith/xml-tool-use-instructions. Tool schema XML is emitted only into user-turn context via zenith_xml_context, zenith_init, reactive result blocks, or host adapter insertion.
	•	Native API tools list is stable and minimal. The only native steady tools are:
	•	read_text_file
	•	write_file
	•	directory
	•	edit_file
	•	search_files
	•	plus the XML gateway tool(s): execute_xml and/or zenith_xml_dispatch
	•	Deferred tools are still registered. read_multiple_files, read_media_file, file_manager, stashRestore, and refactor_batch remain backed by their real handlers, but are surfaced to the model through XML schema injection unless explicitly configured otherwise.
	•	Use a VirtualMcpServer, not duplicated tool rewrites. Existing tool registration files must not be rewritten to a separate registry API. The virtual server preserves the registerTool(name, config, handler) signature.
	•	Use SAX/scanner parsing, not strict XML parser packages. Model output can contain unescaped <, >, &, markdown fences, partial tags, and prose. Strict XML parsers are too brittle.
	•	CDATA JSON is mandatory for complex params. Object/array params are described to the model as stringified JSON inside <![CDATA[...]]>. Formatter may still expose nested shape information for readability, but the call format for complex values is CDATA JSON.
	•	Dispatch is sequential by default. Multiple <tool_call> blocks in one payload execute in document order. This avoids race conditions for filesystem writes.
	•	Errors return soft <tool_result status="error"> blocks. XML syntax errors, validation errors, unknown tools, duplicate call IDs, and handler failures are returned to the model as recoverable tool results.
	•	Session state is bounded. Each session has a rolling 5-turn schema digest LRU, duplicate call-id set, and BM25/BMX retrieval index. Cleanup is mandatory on HTTP DELETE, SSE close, and idle reap.
	•	No new runtime XML dependency. Only dev dependencies for tests/fuzzing are allowed unless a later benchmark proves parser cost is unacceptable.

export class ZenithToolRegistry {
  private readonly tools = new Map<string, RegisteredXmlTool>();

  register(tool: RegisteredXmlTool): void {
    if (!/^[A-Za-z0-9_]+$/.test(tool.name)) {
      throw new Error(`Invalid tool name: ${tool.name}`);
    }
    const existing = this.tools.get(tool.name);
    if (existing) throw new Error(`Duplicate tool registration: ${tool.name}`);
    this.tools.set(tool.name, Object.freeze({ ...tool }));
  }

  get(name: string): RegisteredXmlTool | undefined {
    return this.tools.get(name);
  }

  listAll(): RegisteredXmlTool[] {
    return [...this.tools.values()];
  }

  listNative(): RegisteredXmlTool[] {
    return this.listAll().filter(t => t.nativeVisible);
  }

  listDeferred(): RegisteredXmlTool[] {
    return this.listAll().filter(t => t.xmlVisible && !t.nativeVisible);
  }
}

export class VirtualMcpServer {
  constructor(private readonly registry: ZenithToolRegistry) {}

  registerTool(name: string, config: any, handler: any) {
    this.registry.register({
      name,
      title: config?.title,
      description: config?.description,
      inputSchema: config?.inputSchema,
      outputSchema: config?.outputSchema,
      annotations: config?.annotations,
      enabled: true,
      nativeVisible: false,
      xmlVisible: true,
      handler,
    });

    return {
      enable: () => {
        const tool = this.registry.get(name);
        if (tool) (tool as any).enabled = true;
      },
      disable: () => {
        const tool = this.registry.get(name);
        if (tool) (tool as any).enabled = false;
      },
      update: () => {},
      remove: () => {},
    };
  }
}
Exit criteria:

export async function dispatchSingleXmlCall(opts: {
  call: ParsedToolCall;
  state: XmlSessionState;
  ctx: FilesystemContext;
  timeoutMs?: number;
}): Promise<DispatchResult> {
  const tool = opts.state.registry.get(opts.call.name);
  if (!tool) {
    return errorDispatch(opts, "unknown_tool", `Unknown XML tool: ${opts.call.name}`);
  }

  if (opts.call.callId && !opts.state.reserveCallId(opts.call.callId)) {
    return errorDispatch(opts, "duplicate_call", `Duplicate XML call id: ${opts.call.callId}`);
  }

  const args = validateAgainstInputSchema(opts.call.args, tool.inputSchema);
  const result = await withTimeout(() => tool.handler(args, { ctx: opts.ctx }), opts.timeoutMs ?? 30_000);
  const text = normalizeToolResultText(result);

  const reactiveSchemas = buildReactiveXmlContext({
    state: opts.state,
    toolName: opts.call.name,
    toolResultText: text,
  });

  return {
    call: opts.call,
    ok: true,
    textForRanking: text,
    xml: formatToolResult({
      toolName: opts.call.name,
      callId: opts.call.callId,
      content: text,
      includeReactiveSchemas: reactiveSchemas,
    }),
  };
}

server.registerTool(
  "execute_xml",
  {
    title: "Execute XML tool calls",
    description: "Executes one or more <tool_call> XML blocks against deferred Zenith tools.",
    inputSchema: {
      xml_payload: z.string().max(100_000),
    },
  },
  async ({ xml_payload }) => {
    const state = requireXmlState(ctx);
    const knownTools = new Set(state.registry.listAll().map(t => t.name));
    const parsed = parseToolCalls(xml_payload, knownTools);

    if (parsed.calls.length === 0) {
      return {
        content: [{
          type: "text",
          text: formatToolError({
            toolName: "execute_xml",
            code: "parse_error",
            message: "No valid <tool_call> blocks were found.",
            diagnostic: parsed.diagnostics.map(d => d.detail).join("\n"),
          }),
        }],
      };
    }

    const results = await dispatchXmlCalls({ calls: parsed.calls, state, ctx });
    return { content: [{ type: "text", text: results.map(r => r.xml).join("\n") }] };
  },
);

Cache-Preservation Audit
	•	Static instruction is hash-locked and never includes schema names, params, session IDs, or retrieval output.
	•	Dynamic schema XML is emitted through zenith_xml_context, zenith_init, or reactive tool results, not mutated into a system prompt by Zenith.
	•	Native tools/list is stable after session init.
	•	Deferred schema ranking changes only user-turn context, not native tools.
	•	Last-5-turn digest suppression prevents pointless repeated schema injection.
	•	Gateway tool schemas are stable and always visible when XML is enabled.
	•	Disabling XML happens at startup only and is not a per-turn mutation.
